## Лабораторная работа №13-14. ViewModel и архитектура Android-приложений

### Цель работы:
Познакомиться с архитектурными компонентами Android Jetpack, научиться использовать ViewModel для управления состоянием UI и сохранения данных при изменении конфигурации устройства (поворот экрана, смена темы и т.д.). Разработать игру с подсчётом очков, которая не теряет данные при повороте экрана.

### Описание игры
UnscrambleGame — это простая игра, в которой пользователь должен угадать правильное слово из перемешанных букв.

### Технологии
- Язык: Kotlin
- UI: Jetpack Compose
- Архитектура: ViewModel + StateFlow
- Тестирование: JUnit 4

### Функциональность
- Отображение перемешанного слова
- Поле для ввода ответа
- Проверка правильности ответа
- Увеличение счёта на 20 очков за правильный ответ
- Кнопка "Пропустить" для перехода к следующему слову без начисления очков
- Индикация ошибки при неправильном ответе
- Сохранение состояния при повороте экрана
- Диалог с результатами после отгадывания всех 10 слов
- Кнопка "Играть снова" для перезапуска игры

### Контрольные вопросы

**1. Что такое ViewModel и зачем он нужен?**

ViewModel — это компонент Android Jetpack, который хранит данные UI и не уничтожается при повороте экрана. Без ViewModel все данные (счёт, слово) теряются при изменении конфигурации устройства.

**2. В чём разница между MutableStateFlow и StateFlow?**

MutableStateFlow можно изменять, StateFlow — только читать. Паттерн "backing property" (приватное изменяемое поле \_uiState и публичное неизменяемое uiState) защищает состояние от случайных изменений из UI.

**3. Почему в ViewModel нельзя хранить Context или View?**

Это может привести к утечке памяти — ViewModel живёт дольше Activity, и сохранённая ссылка не даст системе очистить память.

**4. Чем StateFlow отличается от remember и rememberSaveable?**

remember хранит данные только внутри Composable и теряется при повороте. rememberSaveable восстанавливается при повороте, но хранит логику в UI. StateFlow в ViewModel отделяет логику от UI и автоматически обновляет экран через collectAsState().

**5. Что такое data class в Kotlin и зачем он используется для UI State?**

data класс автоматически генерирует equals(), hashCode(), toString() и copy(). Функция copy() позволяет создавать копию объекта с изменением отдельных полей — это удобно для обновления состояния.

**6. Как работает collectAsState() в Compose?**

collectAsState() подписывается на StateFlow и при каждом изменении значения вызывает перерисовку UI.

**7. Зачем разделять код на пакеты data и ui?**

Это поддерживает принцип единственной ответственности: data отвечает за данные и константы, ui_model — за логику и UI. Код становится проще поддерживать и тестировать.

**8. Что происходит с ViewModel, когда приложение полностью закрывается?**

ViewModel уничтожается вместе с процессом. Данные не сохраняются. Для долговременного хранения нужно использовать Room или DataStore.
